# 方案审核检查清单

> 来源：`harness-design-review` skill。用于方案审核时的维度参考，可被其他 skill 引用。

## 四检查面速查

### 1. 范围完整性

- [ ] Scope IN 是否明确描述了要做什么？
- [ ] Scope OUT（Non-goals）是否有 ≥ 2 项，且排除的是"别人可能会以为包含"的合理范围？
- [ ] 关键术语是否有定义？是否存在"快/大/好"等模糊词？
- [ ] 七大思维快速扫描是否有遗漏维度？

### 2. 内部一致性

- [ ] 不同章节对同一概念/术语的描述是否一致？
- [ ] 数据流：输入 → 处理 → 输出 是否前后对得上？
- [ ] 假设列表内部是否有矛盾？
- [ ] 约束条件是否与方案设计冲突？

### 3. 可行性

- [ ] 方案是否在项目技术栈约束内可实现？
- [ ] 外部依赖（API/服务/平台）是否确认可达？
- [ ] 关键假设是否标注了验证方式？
- [ ] 是否存在"假设能跑通就完了"的未验证大跃进？

### 4. 风险覆盖

- [ ] 每个关键假设的失效条件是否分析过？
- [ ] 失效时的影响范围和严重度是否评估？
- [ ] 高风险项是否有缓解措施？
- [ ] 是否有回滚/降级方案？（如适用）

## 常见遗漏模式

| 模式 | 表现 | 检查方式 |
|------|------|---------|
| 只画正常路径 | 所有流程都是 happy path，没有错误分支 | 逐流程问：这个步骤失败怎么办？ |
| 隐式假设 | "数据库会返回正确结果" "用户不会同时操作" | 列出所有"理所当然"的陈述，逐一挑战 |
| 接口未定义 | 只说"A 调用 B"，没说协议/格式/错误码 | 每个箭头问：调用失败返回什么？ |
| 非功能需求缺失 | 没有性能/安全/可观测性要求 | 检查 spec 是否只有功能描述 |
| 范围漂移 | 方案中混入了远超需求的"顺便做" | 逐功能问：不做的后果是什么？如果零后果→该砍 |
