# 常见反模式

> AI 编码时必须避免这些反模式，即使它们看起来"能工作"。
> 从 code review 和流程违规中积累，每个反模式都有真实案例支撑。

## 反模式 1：将非精确匹配的动作词理解为直接执行指令

**案例**：用户说"帮我仔细评估下……优化完善这个方案"。AI 未将"评估"和"优化完善"识别为工程动作词（`harness-entry` 触发边界仅列举了"设计/方案/分析/架构/选型/修bug/实现/开发/写代码/写测试/写文档"），直接跳过了 harness-entry → 分诊的路由，直接读文件→分析→编辑文件。

**为什么是反模式**：
- "评估"≈"分析"，"优化完善"≈"修改文档"，都是明确的工程动作
- 触发边界的关键词列表不可能穷举所有自然语言变体，AI 需要做语义匹配而非字面匹配
- 跳过 harness-entry 意味着跳过了所有 HARD-GATE 纪律检查（分诊深度、模板遵循、知识沉淀……）

**正确做法**：
1. 收到任务后，先做语义判断：用户是否在要求"产生/修改产出物"（文档/代码/设计）？
2. 是 → 触发 harness-entry，走分诊流程
3. 否（纯讨论/提问/澄清）→ 不触发
4. 不确定时 → 触发 harness-entry，让分诊流程决定（宁可多走流程，不可跳过）

**预防措施**：
- `harness-entry/SKILL.md` 触发边界表增加语义判定规则，不依赖关键词穷举
- 增加兜底规则：无法确定是否触发的 → 触发
