# Harness — 项目管理（PMO）

<HARD-GATE>
此 skill 覆盖从 spec/PRD 到可执行任务清单的完整拆解链。流程不可跳过，但用户可在每层拆解后调整。

核心原则：**AI 提案拆解方案，PM 确认后才进入下一层。** 不自动生成不确认的内容。
</HARD-GATE>

## 流程（4 步，从上到下逐层拆解）

```
spec/PRD
    ↓ Step 1（确认后进入 Step 2）
里程碑（时间盒，≤ 4 个）
    ↓ Step 2（确认后进入 Step 3）
史诗（功能主题，2-4 个/里程碑）
    ↓ Step 3（确认后进入 Step 4）
用户故事（P0/P1/P2 优先级排序）
    ↓ Step 4
任务（仅 P0 故事拆解为可执行任务）
```

### Step 1: 读取输入 → 拆解里程碑

**必读文件**：spec（`<project>/.harness/04-changes/<变更>/00-design.md`）或 PRD 文档，及 `docs/architecture/00-overview.md`，及 `docs/patterns/pmo-workflow.md`（拆解流程参考），及 `docs/patterns/pmo-excel-template.md`（Excel 输出模板参考）

**产出**：2-4 个里程碑，每个按以下表格结构展示：

> ### 里程碑总览
>
> | | M1 | M2 | M3 | M4 |
> |---|----|----|----|----|
> | **名称** | | | | |
> | **目标** | | | | |
> | **时间盒** | | | | |
> | **关键交付物** | | | | |
> | **依赖** | — | | | |

**规则**：
- 第一个里程碑（M1）必须是 **MVP 核心**——没有它产品无意义
- 最后一个里程碑（M4 或 M3）是 **体验完善**——可延后但不该缺失
- 里程碑之间是**时间先后**关系，不是并列选项

**展示后等待用户确认，确认后才进入 Step 2。**

### Step 2: 每个里程碑 → 拆解史诗

**产出**：每个里程碑下 2-4 个史诗，按以下结构展示：

> ### M1: [里程碑名称] — 史诗拆解
>
> | | Epic 1 | Epic 2 | Epic 3 |
> |---|--------|--------|--------|
> | **目标** | | | |
> | **覆盖范围** | | | |
> | **涉及模块** | | | |
> | **预估工作量** | | | |

**规则**：
- 史诗是**功能主题**，代表一组相关的用户故事
- 一个史诗内的所有故事向同一个目标收敛
- 史诗之间边界清晰，无重叠

**展示后等待用户确认，确认后才进入 Step 3。**

### Step 3: 每个史诗 → 拆解用户故事（优先级排序）

**格式**：`As a [角色], I want [功能], so that [价值]`

**优先级定义**：

| 优先级 | 定义 | 判断标准 |
|--------|------|---------|
| **P0** | 必须做，否则里程碑无意义 | 高影响 + 高紧迫：没有它用户无法完成核心任务 |
| **P1** | 应该做，重要但不阻塞 | 高影响 + 低紧迫，或低影响 + 高紧迫 |
| **P2** | 可以做，锦上添花 | 低影响 + 低紧迫：做了更好，不做也能用 |

**产出结构**：

> ### M1 — 用户故事总览
>
> #### Epic 1: [史诗名称]
>
> | 优先级 | 用户故事 | 验收标准 |
> |--------|---------|---------|
> | **P0** | As a [角色], I want [功能], so that [价值] | 1. Given/When/Then |
> | **P0** | ... | ... |
> | **P1** | ... | ... |
> | **P2** | ... | ... |
>
> #### Epic 2: [史诗名称]
>
> | 优先级 | 用户故事 | 验收标准 |
> |--------|---------|---------|
> | ... | ... | ... |

**排序依据**：每个故事旁标注影响度（高/中/低）× 紧迫度（高/中/低），优先级按 P0 → P1 → P2 排列

**展示后等待用户确认，确认后才进入 Step 4。**

### Step 4: P0 用户故事 → 拆解为可执行任务

**产出**：仅拆解 P0 故事。P1/P2 留到后续里程碑再做任务拆解。

> ### M1 — 任务拆解（P0）
>
> #### 📋 P0-1: [用户故事标题]
>
> | # | 任务 | 预估 | 依赖 | 验收标准 | 涉及模块 |
> |---|------|------|------|---------|---------|
> | 1 | | | — | | |
> | 2 | | | 1 | | |
>
> #### 📋 P0-2: [用户故事标题]
>
> | # | 任务 | 预估 | 依赖 | 验收标准 | 涉及模块 |
> |---|------|------|------|---------|---------|
> | ... | | | | | |

**规则**：
- 每个任务是一个人半天到一天能完成的粒度
- 依赖列标注前置任务编号（同故事内）或外部依赖（"需先完成 P0-1"）
- 预估用相对值（人天），不编造精确工时

### Step 4.5: 生成交付物（Markdown + Excel）

所有拆解完成后，输出两份交付物：

**产出物 1**：Markdown 规划文档 → `<project>/.harness/04-changes/<变更>/02-project-plan.md`
**产出物 2**：精美 Excel 项目规划表 → `<project>/.harness/04-changes/<变更>/02-project-plan.xlsx`

#### Excel 格式规范

Excel 文件包含以下 Sheet，使用 `openpyxl` 生成：

**Sheet 1: "项目总览"**

| 区域 | 内容 | 格式 |
|------|------|------|
| 标题行 | 项目名 + 生成日期 | 深色背景，白色粗体，合并单元格，行高 36 |
| 里程碑路线图 | M1-M4 横排（名称/目标/时间盒/交付物） | M1=蓝色(#2563eb), M2=绿色(#16a34a), M3=橙色(#ea580c), M4=紫色(#9333ea)，白色字体 |
| 史诗×故事矩阵 | 行=史诗/故事，列=M1-M4，P0/P1/P2 颜色标记 | P0=红色背景(#fef2f2)，P1=黄色背景(#fefce8)，P2=灰色背景(#f9fafb) |
| 统计汇总 | 史诗数/P0/P1/P2/任务数/预估人天 按里程碑横排 | 浅灰背景，粗体数字 |

**Sheet 2-N: "M1-名称", "M2-名称"...（每个里程碑一页）**

| 区域 | 内容 | 格式 |
|------|------|------|
| 顶部 | 里程碑名称 + 目标 + 时间盒 | 对应里程碑主题色背景 |
| 史诗分组 | 每个史诗一个区块，标题行合并单元格 | 浅色背景区分 |
| 用户故事表 | 优先级 / 用户故事 / 验收标准 | P0=红底, P1=黄底, P2=灰底 |
| 任务拆解表 | # / 任务 / 预估(人天) / 依赖 / 验收标准 / 涉及模块 | 表头深色，斑马纹行 |

**通用样式规范**：
- 字体：默认 11pt Calibri，标题 14pt 粗体
- 表头：深色背景 + 白色字体 + 居中
- 边框：细灰线(`#d4d4d4`)
- 列宽自适应内容（中文约 2x 字符宽度）
- 冻结首行（Freeze Panes）
- P0/P1/P2 单元格填充色，优先级列居中

**生成方式**：AI 在 Step 4.5 中，根据已确认的拆解数据，用 Python `openpyxl` 直接生成 Excel。不使用外部模板或脚本文件——数据已在上下文中，直接写 `python3 << 'EOF'` 生成。

**产出路径**：`<project>/.harness/04-changes/<变更>/02-project-plan.md` + `<project>/.harness/04-changes/<变更>/02-project-plan.xlsx`

## 反模式

- "直接全拆完吧，不用每层确认" → 拆一层确认一层，用户可能中途调整
- "把 P1/P2 的任务也一起拆了" → P1/P2 在后续里程碑迭代中细化，不提前过度拆解
- "这个任务大概 3.5 小时" → 用人天做相对估算，不编造精确工时
- "优先级按感觉排就行" → 必须标注影响度 × 紧迫度，让排序有依据

## 跨会话衔接

PMO 是独立步骤，不写入 `pipeline-state.json`。拆解产物（`02-project-plan.md` + `02-project-plan.xlsx`）存入变更目录，下游角色直接读取。

## 触发方式

1. **harness-entry 路由**：用户说"拆任务/排期/里程碑/用户故事"→ 分诊到 harness-pmo
2. **brainstorming 完成后的衔接**：如用户选"生成文档"→ 文档交付后可自然衔接 PMO 拆解
