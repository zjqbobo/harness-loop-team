# 需求分析方法论
> 📌 **适用场景：需求分析、业务建模、方案设计

---

## 🔹 方法一：纲举目张法（五步法）

解决「一页纸需求」问题的完整方法论：

1. **业务价值确认** — 从多个利益相关者角度确认价值
2. **角色梳理** — 明确涉及的所有角色
3. **术语定义** — 建立统一语言，减少误解
4. **主业务流程** — 用泳道图描画主流程
5. **七大思维方式** — 系统展开需求细节

---

## 🔹 方法二：七大思维方式

| 思维方式 | 核心思想 | 应用示例 |
|---------|---------|---------|
| **对称思维** | 考虑正向反向 | 转账↔取消转账，正常↔异常 |
| **边界思维** | 明确上下限 | 单日转账金额(0,30000]，查询时间范围[0,12]月 |
| **多样性思维** | 枚举所有变化 | 转账方式、录入方式、通知方式、账户类型 |
| **生命周期思维** | 完整生命周期 | 发起→处理→结果，各阶段识别字段、功能、外部依赖 |
| **异常思维** | 异常场景处理 | 网络不通畅时的补偿通知机制 |
| **并发思维** | 多用户同时操作 | 两个设备同时操作，防止重复提交 |
| **安全思维** | 系统安全风险 | 请安全专业团队支持 |

## 🔹 方法三：需求质量标准 CRARE

| 标准 | 要求 |
|-----|------|
| **完整性** | 任何没有定义的部分都有可能产生 bug
| **正确性** | 没有正确定义的内容都有可能被错误的实现
| **精确性** | 模糊的词汇会导致误解，不让开发人员武断做决定
| **易读性** | 文档应该容易阅读，一图胜千言
| **一致性** | 采用相同语法结构、工具、文档格式
| **柔软性** | 需求文档应该容易被编辑
| **可测性** | 需求描述应该可以通过测试来验证

## 🔹 方法四：业务建模四大方法

### 4.1 流程建模
**目标：识别共性业务，抽取通用流程，设计可变点

**三原则：**
1. 统一命名规范：动词在前，名词在后
2. 使用业务术语：不用技术术语
3. 包含跨系统流程

**两种方法：**
- 记叙文六要素法：自下而上梳理
- 自顶向下分解法：复杂业务域

**产出：阶段 → 活动 → 任务 → 步骤 + 业务规则 + 可变点

### 4.2 领域建模
**目标：划清业务边界，识别可复用的领域对象和聚合

**概念层级：**
```
域 → 子域 → 限界上下文 → 聚合 → 聚合根 → 实体 / 值对象
```

**产出：领域概念模型图 + 子域边界划分图

### 4.3 业务身份建模
**目标：平台同时给多个业务提供服务时提供差异化个性化

**四要素：** 业务身份ID、名称、要素定义、识别解析规则

**产出：业务身份矩阵表 + 识别规则说明

### 4.4 能力建模
**目标：构建可复用的能力层级，支撑业务快速编排

**五级模型：**
```
基础能力 → 扩展点 → 第一类能力组件 → 第二类能力组件 → 解决方案
```

**产出：能力层级图 + 基础能力清单 + 扩展点定义表
