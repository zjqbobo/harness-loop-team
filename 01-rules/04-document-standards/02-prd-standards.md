# PRD 产品需求文档标准
> 📌 **适用场景：生成任何产品需求文档（PRD）时必须遵守以下规范**

---

## 🔴 HARD RULE（最重要，Blocking Gate）

**生成 PRD 前，第一步必须读取标准模板文件并严格遵循其完整章节结构。**

```
模板文件: 09-templates/PRD标准模板_v2.0.md
         ~/.claude/templates/PRD标准模板_v2.0.md（安装后）

规则:
1. ⛔ 禁止凭记忆生成 PRD，必须先 Read 模板文件
2. ⛔ 禁止只遵循主章节、忽略子章节
3. ⛔ 禁止缺省任何一个子章节（8大章节 × N子章节 = 40+子章节）
4. ✅ 生成后必须逐章节核对，缺失任何子章节视为不合格
```

---

## 🔴 强制执行规则
1. **必须按 8 大章节结构输出，禁止缺省章节**
2. **禁止混入技术方案内容**（数据表设计、接口定义、架构图属于技术方案，不属于 PRD）
3. **每个功能模块必须包含完整 11 个子章节结构**
4. **必须使用精益需求方法论填充内容**
5. **必须使用业务建模四大方法分析业务**
6. **生成前必须先确认需求边界和用户故事**
7. **生成前必须先 Read `09-templates/PRD标准模板_v2.0.md`（或 `~/.claude/templates/PRD标准模板_v2.0.md`），严格对齐其所有子章节**

---

## 📋 标准模板章节结构
生成 PRD 文档必须包含以下全部章节结构：

---

### 📌 文档头
```markdown
# [产品名称] 产品需求文档 (PRD)

| 文档编号 | PRD-XXXX-001 | 文档密级 | 内部 |
| --- | --- | --- | --- |
| 归属项目 | | 归属部门 | |
| 编写人 | | 编写日期 | YYYY-MM-DD |
| 审核人 | | 审核日期 | |
```

---

## 一、文档概览
必须包含以下**4个子章节**：

| 子章节 | 内容要求 |
|-------|---------|
| 1.1 修订记录 | 版本号、修订人、修订日期、修订描述 |
| 1.2 阅读对象 | 角色+关注重点（产品、设计、研发、测试、运营、项目经理） |
| 1.3 术语与名词解释 | 术语缩写、全称、说明 |
| 1.4 参考文档 | 文档名称 + 链接/位置 |

---

## 二、产品概述
必须包含以下**6个子章节**：

| 子章节 | 内容要求 |
|-------|---------|
| 2.1 项目背景 | 现状问题、痛点、业务场景描述 |
| 2.2 产品目标 | 6个维度：核心定位、商业价值、用户价值、成功指标(KPIs) |
| 2.3 目标用户 | 用户画像：角色、画像、核心使用场景 |
| 2.4 产品范围 | 本期包含、本期不包含（明确范围边界） |
| 2.5 产品路线图 | 版本、时间范围、核心功能概要（V1.0 / V2.0） |
| 2.6 假设与约束 | 业务假设、技术约束、合规要求、资源限制 |

---

## 三、全局规范
必须包含以下**6个子章节**，每个子章节还需包含其内部细分：

| 子章节 | 内容要求 |
|-------|---------|
| 3.1 功能清单 | 一级功能、二级功能、三级功能、功能描述（完整树状结构） |
| 3.2 业务架构 | 3.2.1 流程模型（泳道图+可变点设计）<br>3.2.2 领域模型（实体关系+聚合根边界）<br>3.2.3 业务身份模型（矩阵表+识别规则）<br>3.2.4 能力模型（基础能力→扩展点→解决方案层级） |
| 3.3 全局规则矩阵 | 3.3.1 权限矩阵（功能×角色）<br>3.3.2 字段校验矩阵<br>3.3.3 配置唯一性判定矩阵<br>3.3.4 并发冲突处理矩阵 |
| 3.4 全局交互规则 | 页面加载、网络异常、消息通知、数据缓存、返回逻辑、二次确认 |
| 3.5 全局数据规则 | 优先级定义、数据同步、字典定义、软删除规则 |
| 3.6 页面设计规范 | 布局规范、操作提示、状态说明、按钮规范 |

---

## 四、功能需求
按模块逐一展开，**每个模块必须包含完整11个子章节**：

| 子章节 | 方法论 | 内容要求 |
|-------|-------|---------|
| 4.X.1 功能概述 | - | 功能定位、核心价值 |
| 4.X.2 前置/后置条件 | - | 前置条件、后置条件 |
| 4.X.3 用户场景 | **故事线法** | 3个+用户故事 + 故事线验证 |
| 4.X.4 主要流程 | - | 核心流程的步骤拆解 |
| 4.X.5 分支/异常流程 | **七大思维方式** | 对称思维、边界思维、多样性思维、异常思维、并发思维、生命周期思维、安全思维覆盖 |
| 4.X.6 页面原型 | **自动生成后填充** | 页面结构图 + 交互说明 |
| 4.X.7 信息与显示规则 | - | 字段展示形式、字数限制、格式要求 |
| 4.X.8 交互规则 | - | 各元素的交互行为、状态说明 |
| 4.X.9 数据规则 | - | 字段类型、必填、校验规则、默认值 |
| 4.X.10 异常与边界情况 | - | 所有异常场景的处理方式 |
| 4.X.11 验收标准 | **Given-When-Then** | 至少7条，覆盖P0-P2优先级 |

---

## 五、非功能需求
必须包含以下**6个子章节**：

| 子章节 | 内容要求 |
|-------|---------|
| 5.1 性能需求 | 各操作的响应时间、并发量支持（8项指标） |
| 5.2 安全性需求 | 数据加密、访问控制、防重放、注入防护、审计（6项） |
| 5.3 兼容性需求 | 浏览器、分辨率、后端版本、数据库（4类） |
| 5.4 可用性需求 | 系统可用率、故障恢复、数据保留、降级策略（4项） |
| 5.5 可维护性需求 | 配置化、热插拔、日志标准化、健康检查（4项） |
| 5.6 监控与告警需求 | 核心指标监控、异常告警、慢日志告警（4项） |

---

## 六、数据需求
必须包含以下**2个子章节**：

| 子章节 | 内容要求 |
|-------|---------|
| 6.1 数据统计需求 | 6项以上核心指标：定义、计算公式、统计维度、看板 |
| 6.2 数据导出需求 | 导出内容、格式、频率、范围限制、使用角色 |

---

## 七、项目规划
必须包含以下**4个子章节**：

| 子章节 | 内容要求 |
|-------|---------|
| 7.1 里程碑计划 | 里程碑、时间、交付物、验收标准（至少7个里程碑） |
| 7.2 研发计划表 | 阶段、时间范围、负责人、交付物（7个以上阶段） |
| 7.3 风险分析 | 风险描述、类型、等级、应对预案、负责人（5项以上） |
| 7.4 效益成本分析 | 投入资源、预期收益、ROI评估 |

---

## 八、交付与上线
必须包含以下**5个子章节**：

| 子章节 | 内容要求 |
|-------|---------|
| 8.1 交付物清单 | 完整交付物列表及责任人 |
| 8.2 上线方案 | 灰度策略、比例、回滚机制、分步计划、兼容方案 |
| 8.3 运营计划 | 培训、推广、引导方案、上线后值守 |
| 8.4 上线后评估 | 数据回顾、用户反馈、性能评估、后续迭代方向 |
| 8.5 附录 | 相关文档链接 |

## 🔍 方法论应用要求
所有 PRD 生成必须应用以下方法论：

### 1. 精益需求分析核心原则
- 纲举目张法（五步法）
- 七大思维方式（对称、边界、多样性、生命周期、异常、并发、安全）
- 需求质量标准（完整性、正确性、精确性、易读性、一致性、柔软性、可测性）
- CRARE 七项标准

### 2. 业务建模四大方法
- **流程建模**：统一命名、记叙文六要素、可变点设计
- **领域建模**：聚合根、实体、值对象、上下文边界
- **业务身份建模**：多业务场景差异化服务、身份识别四要素
- **能力建模**：五级能力层级、扩展点设计、基础能力三原则
