# 原型生成工作流规范
> 📌 **适用场景：生成 UI 原型、界面设计、交互稿、mockup

---

## 🔴 强制执行规则
**生成 UI 原型前必须先询问用户选择，不得直接默认生成。

---

## 1️⃣ 原型工具选择（必须询问）

### 询问话术
当用户表达以下意图时，**第一步先询问：
> "请问您希望使用哪种方式生成原型？
> 🎨 Figma MCP 或 💎 ui-ux-pro-max 插件"

### 触发场景
- "生成原型"
- "做个 UI"
- "设计界面"
- "画个原型"
- "创建 mockup"

---

## 🚫 绝对禁止规则
> ❌ **禁止用文字描述代替原型截图**
> 
> 所有 PRD 的「页面原型」章节必须贴真实的 UI 截图，不允许出现：
> - "此处省略原型，后续补充"
> - "页面布局如图所示"（但没有图）
> - 用 ASCII 字符画、纯文字排版代替截图
> - 用 mermaid 流程图代替 UI 原型
>   
> ✅ 正确做法：生成 HTML 原型 → 自动截图 → 嵌入文档

---

## 2️⃣ PRD 原型自动化工作流

生成 PRD 文档时，自动执行以下流程，无需用户额外指令

### 阶段1：PRD 文档内容生成
1. 按精益需求方法论完成 PRD 内容编写
2. 确保所有功能模块都包含「页面原型」章节
3. 标记清楚每个需要原型的位置

### 阶段2：高保真 HTML 原型生成
1. 生成完整的交互式 HTML 原型（mockup）
2. 包含所有页面：流程列表、配置页、详情页等
3. 实现页面间可导航切换
4. 保存为 `{项目名}_原型.mockup.html`

### 阶段3：自动截图
自动执行 Node.js 脚本完成截图：
1. 启动本地 HTTP 服务器
2. 使用 Playwright + Chrome 浏览器
3. 遍历所有页面截图
4. 保存到 `pbc-images/` 目录

### 阶段4：自动填充文档
同时填充两种格式：

#### Markdown 格式
- 自动将图片插入到对应原型章节
- 使用 `![页面名](pbc-images/xxx.png)` 语法
- 图片 alt text 必须是页面的功能描述（便于无障碍阅读）

#### Word 格式
- 自动将图片插入到 Word 文档对应位置
- 每张图片宽度固定 6.2 英寸
- 图片下方添加居中图题说明（如 "图 4-1 流程列表页面"）
- 图片必须保持原始清晰度，禁止压缩导致模糊

#### 📸 截图质量规范
| 要求 | 标准 |
|------|------|
| 分辨率 | 宽度 ≥ 1200px |
| 格式 | PNG 格式（禁止 JPG，避免压缩失真） |
| 完整性 | 必须包含完整浏览器窗口（含标题栏）或完整页面 |
| 清晰度 | 文字无锯齿、按钮无变形 |
| 一致性 | 所有截图窗口大小一致，缩放比例统一 |
| 禁止内容 | 截图不得包含浏览器收藏夹、扩展图标、桌面水印 |

## 3️⃣ 技术实现说明

### 截图脚本
- 脚本位置：项目根目录 `capture-prototype.js`
- 使用 `http-server + playwright
- 使用系统 Chrome 浏览器（无需下载浏览器

### 图片插入脚本
- 脚本位置：项目根目录 `insert-images-to-docx.py`
- 使用 `python-docx` 库
- 自动定位关键词位置插入图片

## 5️⃣ 禁止行为

❌ 禁止只生成文字内容不生成原型
❌ 禁止只生成原型不截图不自动填充文档
❌ 禁止要求用户手动截图或手动插入图片
❌ 禁止交付不含图片的文档作为最终交付物
