# 开发流程规范总览

> 📌 本节规范开发全生命周期流程，所有开发任务遵循本规范

---

## 🔹 核心流程

```
需求分析 → 技术方案 → 编码实现 → 代码评审 → 单元测试 → 集成测试 → 上线发布 → 线上监控
```

---

## 🔹 阶段定义

| 阶段 | 输入 | 输出 | 关键检查点 |
|------|------|------|-----------|
| 需求分析 | 业务需求 | PRD 文档、用户故事 | 需求是否明确、范围是否清晰 |
| 技术方案 | PRD 文档 | 技术方案、架构设计 | 是否遵循五层架构、是否可扩展 |
| 编码实现 | 技术方案 | 代码、配置 | 编码规范合规、8层模板覆盖 |
| 代码评审 | 代码 | 评审意见 | 无严重问题、安全漏洞 |
| 单元测试 | 代码 | 测试用例、报告 | 覆盖率 ≥ 80%、全通过 |
| 集成测试 | 代码 | 集成测试报告 | 上下游接口正常、满覆盖 |
| 上线发布 | 测试报告 | 部署记录 | 灰度策略、回滚方案就绪 |
| 线上监控 | 部署 | 监控告警 | 错误率/时延/吞吐达标 |

---

## 🔹 分支策略

| 分支类型 | 命名规则 | 用途 |
|---------|---------|------|
| 主干分支 | main / master | 生产代码，受保护 |
| 开发分支 | develop | 日常开发集成 |
| 功能分支 | feature/{需求编号}-{描述} | 新功能开发 |
| 修复分支 | bugfix/{问题编号}-{描述} | bug 修复 |
| 热修复 | hotfix/{问题编号}-{描述} | 生产紧急修复 |
| 发布分支 | release/{版本号} | 版本发布准备 |

---

## 🔹 代码提交规范

```
<type>(<scope>): <subject>

type: feat|fix|refactor|docs|test|style|chore|ci
scope: 模块名
subject: 50字内的简要描述
```
